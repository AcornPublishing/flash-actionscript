Button.onRollOut

instanceName.onRollOut = functionName;

 

Where instanceName is the instance name of a Button, and functionName is a function containing the code you wish to execute every time the onRollOut button event occurs.

 

This event handler is compatible with Flash 6 or later. Previous versions of Flash did not support the use of function references as call-backs, so this form of event handler script was not supported. For adding button events to buttons in Flash 5 and previous versions, see the on action.

Description

Flash buttons generate events every time a mouse interaction occurs with respect to a button. For something to happen when an event occurs, you must define a script that you want to execute in response to the event, and this script is called the event handler. To understand the operation of Button.onRollOut requires two things; an understanding of the event and how to set up the event handler.

The event

The onRollOut event will be invoked when the mouse moves out of the button-hit area. The user does not have to have clicked on the button, although the SWF does have to have browser focus.

 

The onRollOver event will also occur during the process that causes the onRollOut. It will occur when the mouse enters the button hit area. See also Button.onRollover.

The event handler

The Button.onRollOut event handler definition is used to identify the code that will act as the event handler when the onRollOut event occurs. This code is defined as a function. When a function is used as an event handler, it is known as a call-back, a term inherited from JavaScript.

 

·       If no call-back is defined, the onRollOut event will do nothing.

·       If a call-back is defined, the function will run once every time the onRollOut event occurs.

 

 

Code

 

 

Notes

 

function myFunction() {

   trace("I have been rolled out ");

}

my_btn.onRollOut = myFunction;

 

 

This script defines a function myFunction() as the call-back for this event.

 

 

my_btn.onRollOut = function() {

   trace("I have been rolled out");

};

 

 

This script defines an anonymous function as the call-back.

 

 

my_btn.onRollOut = undefined;

 

This script un-defines a previously defined call-back. You can also set it to null to achieve the same effect.

 

Examples and practical uses

The onRollOut is most commonly used as a trigger rather than a button. For example, you may want an animation to run when the mouse is over it. You can detect whether the mouse is or is not over a particular area by using the onRollOut and onRollOver events. See also Button.OnRollOver. When detecting the mouse in this way, you may not want the user to see the button that is making the rollover detections. To allow this, you can create invisible buttons that:

 

·         Cannot be seen.

·         Do not turn the mouse pointer into the hand cursor when the mouse is over them.

·         Cannot be selected via the tabOrder (i.e. the button cannot gain input focus).

 

See the example button.onrollout.fla and button.onrollout.swf , which shows the use of both Button.onRollOver and Button.onRollOut.

Tips and precautions

This Button event must be in a SWF that has browser focus before the event will start to be detected. This means that the SWF containing the button must have been clicked at least once since being opened.

 

To create an invisible button (as mentioned above) that always stays invisible, your button must be configured to:

 

·       Never accept input focus, otherwise the button may become highlighted. To do this set Button.tabEnabled = false.

·       Stop the mouse becoming the hand cursor when the mouse is over the button. To do this, you need to set the Button.useHandCursor = false.

 

The discussion for this button event refers to the concepts of browser focus and input focus. Readers who are not sure about the difference between these two types of focus may find the notes below useful.

Browser focus

Any frame or separate media stream embedded within a HTML document has to have browser focus before it can detect keyboard inputs. This is an intentional feature to safeguard the user. Without it, it would be possible for a user to be inputting data into a HTML document containing a form, whilst unknown to him, a separate element on the same page (such as a banner advert) could be trapping the same keyboard inputs and using them without the users consent. To combat this, no document or embedded media stream can have browser focus unless it is specifically clicked on. A Flash SWF will not be able to respond to any button clicks unless it has this browser focus, and this is gained by making the user click on the SWF at least once.

Input Focus

Within the Flash SWF, there are a number of elements that may be able to listen to and accept user key-presses. Normally, the user is filling in a form (or similar), and they want to access the elements (typically textFields with associated "accept" buttons) separately and in some kind of sequence. This sequence is called the tabOrder, and the current element that is able to listen to the user key-presses is said to have input focus.

 

Input Focus can be captured by an onscreen textField, Button, or movie clip by:

 

·       The user selecting a textField by clicking on it (irrespective of whether the textField is actually set up to allow user input  If it isnt, the Input Focus will still go to the textField and the appropriate events will occur, but the key-presses will not be echoed in that textField )

·       A button/movie clip or textField has its tabEnabled property as true (this is the default case) and the user has hit the TAB key until the button/movie clip is highlighted (this highlighting is a yellow bounding box by default) or the textField appears selected, with the text entry cursor flashing inside it. The Button /MovieClip /textField events will now have input focus, and the key-press stream will generate the appropriate events within the Button /MovieClip /textField.
