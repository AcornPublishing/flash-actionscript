!= Inequality

b != a;

 

Where a and b are expressions that can be evaluated into a number, Boolean or string. If b is unequal to a, true is returned. If b is equal to a, false is returned.

 

Compatible with Flash 5 or later. There are no known issues with any version of Flash that supports this operator.You should however be aware that type is ignored in the comparison; the number 1 and the string "1" are equal for example. You should use strict inequality (!==), or more usually strict equality (===) if you want to consider type as well as value.

Description

This operator tests for inequality in two values, a and b, and gives true if they are unequal, and false if they are equal.

 

Also, note that, (b != a) = (a != b) = !(b == a)

(b <> a) is the deprecated form, and should be avoided.

Two valueless (undefined) expressions are assumed to be equal, hence d= false.

If any of the two expressions compared is undefined, they are assumed to be unequal, hence e= true.

 

Examples and practical uses

The != operator can be used to give the opposite result of equality (==). It is used extensively in if... actions. See the example (inequality).fla and (inequality).swf for an example of using != with several text fields that must each be filled with a correct value. By comparing the entered text (via textField.text ) with the required string (which is the variable name of each textfield, textField.variable ) with !=, we can find the number of wrong answers. Comparison via == would give us the number of correct answers.

Tips and precautions

As described below:

Code Optimization

When creating if... logic, it is best to use != or == based on which is most likely to occur:

 

if ( a!= b) {

   // do code A

} else {

   //do code B

}

 

The above code is optimized for performance if it is more likely that a and b are unequal. If it is more likely that a and b are equal, do this:

 

if (a == b) {

   // do code B

} else {

   //do code A

}

Effects of rounding errors in comparison operators

When comparing two numbers, take care to remember that value comparisons are made to the full accuracy that Flash uses. This means that 124 and 124.000001 are unequal. You should avoid comparing floating point numbers because rounding errors will tend to make them unequal due to accuracy errors. Consider using Math.float(), Math.ceil(),or Math.round() to round floating point numbers to the nearest integer (or required number of decimal places) before any comparisons take place. If you wanted to compare to a fixed number of decimal places, multiply up before rounding. The code below compares cash values to the nearest cent.

 

function compareCashValues(value1, value2) {

   var value1, value2;

   value1 = Math.round(value1*100);

   value2 = Math.round(value2*100);

   return (value1 != value2);

}

// a and b are in dollars and cents, but one has a rounding error...

a = 123.45;

b = 123.45001;

c = (a != b);

d = compareCashValues(a, b);

trace(c);

trace(d);

 

c will take into account the rounding error of $0.00001 and give an incorrect result, whereas d is correct to the nearest cent.

 

Rounding errors are a fact of life when you are working with motion graphics; the placement of elements on the stage is not (and doesn’t have to be) accurate to the full range of numbers that he scripting can handle.

 

If you want to check for a particular value such as x != 63.5 and x is increasing, you may want to consider using the greater than or equal operator instead, because that will check for a range rather than a definite value, and will give you the required logic even if x has rounding errors.

Comparing strings

When comparing strings, you should be aware that all string comparisons are case sensitive and include leading and lagging spaces. When taking inputs from the user or elsewhere for comparison, make sure that your input code ensures that all strings are the same case and leading/lagging spaces are deleted before comparisons take place. See the String object to see how to do this.

Comparing undefined values

Errors can creep into your code if both or either of the expressions compared is undefined. See the table in the description section to see how this affects your result. Ensure that both expressions are defined before the comparison if you are getting odd results.
