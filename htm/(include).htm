#include

#include "myFile.as";

 

Where myFile.as is the name and path to an.as script file.

 

Compatible with all versions of the Flash player, although only the Flash 5 authoring environment and above support it (this action is used by the Flash authoring environment as a code management feature, and compiled SWF code does not incorporate it.

Description

This action (although the #include action is really a "SWF compiler directive"rather than a true action) is used during runtime to include an external script within the current script. The script will be inserted in place of the #include during compilation. If the script file is not found during the compilation process, a syntax error will be raised.

 

#include is used during compilation time only. The action is replaced by the script it points to during compilation, meaning that #include does not exist in the final SWF. This means that compatibility with previous versions of the Flash player is not an issue; only the authoring environment ever sees this action.

Flash does not discriminate between a true.as file and any other file that you may specify, and the specified file is always loaded in. This is a feature that may sometimes cause Flash to crash. You should always use the.as extension to avoid loading an incorrect file type.

 

Examples and practical use

The #include action is useful in the management of common ActionScript code modules. By building up a library of fully tested modules containing often used code sections (particularly functions), the re-use of fully tested working code via a simple import workflow is possible. When creating object classes or prototypes however, it may be better to use components and the #initclip... #endinitclip structure.

 

Using #include is also a good way of keeping your code separate from the graphic assets. You can do this by adding an #include for all your major code sections (in Flash 6 it is actually possible to write all your code on frame 1 of every timeline, so this is more relevant for that version. In Flash 5 and earlier, the code is usually more distributed throughout the FLA, and this does not apply). Then, if the client asks for the graphic assets, you can send them over as the FLA without necessarily having to send the code that drives it. Something that beginners may see as an odd thing to do, but in the strange world of multimedia contracts and copyrights, it is sometimes a useful bargaining point, as you will soon find out!

 

Finally, you can use the #include file as a "key", whereby the #include file sets up some critical code (usually initialisation) that the main FLA will not work without. You can do this in both Flash 5 and 6.

 

To create a.as file you:

 

1.    Write your code in the actions window.

2.    Then in the actions window pull-down menu, select Export As File...

3.    Finally, save the file with the required folder and directory.

 

.as files can also be loaded directly without using #include by using the Import From File... option in the same pull-down menu. This has the advantage of allowing you to customize the imported scripts. See the example (include).fla and (include).swf .

Tips and precautions

The Authoring environment does not check the syntax of the imported file, only that the name and path of the file to be imported does point to a file that exists. Any scripting errors that fail silently when executed by the Flash player will not be detected. You should therefore check all scripts destined for export as.as files for syntax before you save them.

 

The.as file is only used during actual compilation, and the final SWF file does not need to refer to it to run. Changes made to a.as file will not be reflected in the SWF until it is recompiled via the Flash test movie/test scene or publish options.
