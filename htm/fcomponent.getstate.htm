FComponent.getState

(This method is applicable to FRadioButton.getState. )

 

a = myComponent.getState();

 

Where a is equated to a Boolean value that signifies whether the radio button instance myComponent is currently selected (true ) or unselected (false ).

 

This method is compatible with Flash 6.

Description

The radio button has two different types of output:

 

1.       The currently selected radio button in the radio button group. This value tells you which of several related (or grouped) radio buttons is selected. This is usually the output you are interested in knowing. See also FComponent.getValue(), as applied to a radio button.

2.    The state of a particular radio button instance. This value tells you whether a specific radio button within a group is currently selected or not. This value is returned via the getState method. This output is not usually needed because the getValue method is more relevant.

 

The reason why getValue is more relevant is as follows. If you have a single radio button, you can select it by clicking on it, but you cannot unselect it. If you have radio buttons within a group, then you select any radio button by clicking on it, and you can now unselect it by clicking on any other radio button within the same group. This means that radio buttons are usually grouped, and your code will treat the radio button group as if it were a single component; you do not need to consider individual radio buttons.

Examples and practical uses

There are two standard ways of reading and using a change of state in a radio button;

 

1.       Event driven via the change handler. Here, you read the change directly via the event it generates. The script that actually does something about the change is also included in the change handler.

2.    Polling on demand. Here, you read the state when you need it rather than when the state changes. Although many beginners choose this way (it is simpler to set up and understand because you do not have to set up the event handler) it is possible to miss one or more changes in state.

 

We will therefore only look at the first option. See the example fcomponentgetstate1.fla / fcomponentgetstate1.swf. This example shows how to read the state of three grouped radio buttons. The change handler (radioHandler ) simply outputs the current state of each of the three buttons to text fields, but in practice, the handler would also include code that actually does something with the new values:

 

function radioHandler() {

    radio1_txt.text = radio1.getState();

    radio2_txt.text = radio2.getState();

    radio3_txt.text = radio3.getState();

}

 

 

This code has a number of disadvantages, and they include:

 

·       The code is not general because the getState method requires a definite instance name.

·       You have three button values to look at. You could define a separate handler for each button, but that would leave you in a worse condition, because then you would have three event handlers, plus the requirement to add more if you ever added more radio buttons!

 

Now look at fcomponentgetstate2.fla / fcomponentgetstate2.swf. This doesn’t actually use getState at all, but the related method FComponent.getValue. As you can see, the event handler this time round has a number of advantages:

 

·       It is general because it uses the general argument component, which is the component that raised the onChange event.

·       It is shorter.

·       It only returns one value: the name of the button that is selected.

·       It would not have to change if you added more radio buttons to the group.

·       It can be used for more than one radio button group because of its generality

 

function radioHandler(component) {

    group_txt.text = component.getValue();

}

 

 

As you have probably by now guessed, getState is a very poor relation to the more general method FComponent.getValue. There is only one (very rare) situation when you would use getState instead, and that is if you had a single radio button on its own (and even then, this case will usually be due to a mistake on the designers part; you would be better off using a push button, check box or standard Flash Button object rather than a radio button).

Tips and precautions

Short and sweet: Use FComponent.getValue instead!
